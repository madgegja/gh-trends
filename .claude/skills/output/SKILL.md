# Full Output Enforcement

## Baseline

모든 작업을 프로덕션 수준으로 취급. 부분 출력 = 깨진 출력.
간결함이 아닌 완전성에 최적화.

## 금지 패턴 (Hard Failure)

### 코드 블록 내:
- `// ...`, `// rest of code`, `// implement here`, `// TODO`
- `/* ... */`, `// similar to above`, `// continue pattern`
- `// add more as needed`, 생략을 나타내는 `...`

### 산문 내:
- "계속하려면 알려주세요"
- "자세한 내용은 요청하시면..."
- "간결함을 위해", "나머지도 같은 패턴"
- "나머지는 비슷하게", "등등" (실제 내용 대체 시)

### 구조적 편법:
- 전체 구현 요청에 스켈레톤만 출력
- 첫 번째와 마지막 섹션만 보여주고 중간 생략
- 반복 로직을 예시 1개 + 설명으로 대체
- 코드 작성 대신 "이런 코드를 작성해야 합니다" 설명

## 실행 프로세스

1. **Scope** — 전체 요청 읽기. 기대되는 산출물 수 카운트. 그 숫자 고정.
2. **Build** — 모든 산출물 완전히 생성. 부분 초안 없음.
3. **Cross-check** — 출력 전 원래 요청 재확인. 산출물 수 비교. 빠진 것 있으면 추가.

## 토큰 한계 도달 시

- 남은 섹션 압축 금지
- 결론으로 건너뛰기 금지
- 깔끔한 중단점까지 전체 품질로 작성 (함수 끝, 파일 끝, 섹션 끝)
- 다음으로 종료:

```
[PAUSED — X/Y 완료. "continue" 입력 시 다음 섹션부터 재개]
```

continue 시 중단점에서 정확히 재개. 요약/반복 없음.

## Quick Check

최종 응답 전 확인:
- [ ] 위 금지 패턴이 출력 어디에도 없는가?
- [ ] 사용자가 요청한 모든 항목이 존재하고 완성되었는가?
- [ ] 코드 블록에 실제 실행 가능한 코드가 있는가? (설명 아님)
- [ ] 공간 절약을 위해 축약한 것이 없는가?
